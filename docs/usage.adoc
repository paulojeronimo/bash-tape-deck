= Usage

[[usage-via-scripts]]
== Via scripts

=== Play

Run steps 1 (default):

[,bash]
----
./play.sh samples/fibonacci
----

Run a specific steps file:

[,bash]
----
./play.sh samples/fibonacci 2
----

This expects `samples/fibonacci/steps.2.sh` to exist.

Metadata for each steps file is loaded from `samples/fibonacci/steps.yaml` (multi-document YAML) using `yq`.

Optional lifecycle hooks can be placed next to the steps files:

* `steps.begin.sh`: runs once before the tape starts
* `steps.end.sh`: runs only when the tape is explicitly ejected

These hooks run in the same working directory used by the tape itself. They are useful for setup and cleanup tasks such as temporarily changing local configuration and restoring it afterward.

The player also exports a small set of environment variables so tape scripts can discover their execution context without hardcoded paths:

* `BASH_TAPE_ID`: stable tape identifier derived from the steps directory
* `BASH_TAPE_DIR`: base directory for {BashTapeDeck} internal data in the current working directory
* `BASH_TAPE_STATE_DIR`: tape-specific directory for hook/script state files
* `BASH_TAPE_LOG_FILE`: JSONL event log for the tape
* `BASH_TAPE_STEPS_DIR`: absolute path to the tape directory
* `BASH_TAPE_STEPS_NUMBER`: current `steps.N.sh` number being played or rewound
* `BASH_TAPE_RUN_WORKDIR`: effective working directory used by the tape

When a tape needs to persist auxiliary files, `BASH_TAPE_STATE_DIR` is the intended location.

Lifecycle flow:

* `steps.begin.sh` runs automatically when the tape starts.
* `s` stops the player without running `steps.end.sh`.
* `e` ejects the tape and runs `steps.end.sh`.
* If `steps.end.sh` exists and you stop without ejecting, {BashTapeDeck} warns that cleanup was skipped and the environment may remain inconsistent.

By default, step commands run in `build/` at the project root.
You can override this by setting `BASH_TAPE_RUN_WORKDIR`:

[source,bash]
----
BASH_TAPE_RUN_WORKDIR=/tmp/demo ./play.sh samples/fibonacci 2
BASH_TAPE_RUN_WORKDIR=/tmp/demo ./rewind.sh samples/fibonacci 2
----

You can also export the variable once before running multiple commands:

[source,bash]
----
export BASH_TAPE_RUN_WORKDIR=/tmp/demo
./play.sh samples/fibonacci 2
./rewind.sh samples/fibonacci 2
unset BASH_TAPE_RUN_WORKDIR
----

If you export the variable for temporary testing, it is a good idea to remove it afterward with `unset BASH_TAPE_RUN_WORKDIR`.

TIP: `BASH_TAPE_RUN_WORKDIR` is also honored by `./docker/run.sh play.sh ...` and `./docker/run.sh rewind.sh ...`.
See <<usage-via-docker,Via Docker>> for the wrapper behavior and default working directory.

=== Rewind

Run default reset logic (step 1):

[,bash]
----
./rewind.sh samples/fibonacci
----

Run step-specific reset logic:

[,bash]
----
./rewind.sh samples/fibonacci 2
----

This executes `samples/fibonacci/steps.2.rewind.sh`.

Each `steps.N.rewind.sh` owns the reset strategy required by `steps.N.sh`.

== Via Docker

Build the engine image:

[,bash]
----
./docker/build.sh
----

Build a modular image for a specific tape:

[,bash]
----
./docker/build.sh samples/fibonacci
----

Rebuild the engine first, then build the tape image:

[,bash]
----
./docker/build.sh --all samples/fibonacci
----

The `--all` option is only meaningful when a tape directory is provided.
It forces a rebuild of the engine image before building the tape image.

This creates `bash-tape-deck:fibonacci` with:

* engine scripts (`play`, `rewind`, helpers)
* steps from `samples/fibonacci`
* only sample-specific packages declared in `samples/fibonacci/docker.extend.sh`

Run the modular image directly:

[,bash]
----
docker run --rm -it bash-tape-deck:fibonacci play
docker run --rm -it bash-tape-deck:fibonacci play 2
docker run --rm -it bash-tape-deck:fibonacci rewind
docker run --rm -it bash-tape-deck:fibonacci rewind 2
----

Without a bind mount, generated files are created under `/workspace` inside the container and are lost when the container exits.
This remains the default internal working directory when running the sample image directly with `docker run`.

To control where the working files are created on the host, mount a directory to `/workspace`:

[,bash]
----
docker run --rm -it -u "$(id -u):$(id -g)" -v "$PWD:/workspace" \
  bash-tape-deck:fibonacci play
----

In this example, files such as `fibonacci-lab` are created in the current host directory.

When using `./docker/run.sh play.sh ...` or `./docker/run.sh rewind.sh ...`, the wrapper mounts a host directory as `/workspace` automatically.
By default, that host directory is `build/` in the project root.

This means that, unless you override it, files generated by the wrapper are created under `build/` at the project root.

With the wrapper, you can use the same variable name to override that host-side working directory:

[,bash]
----
BASH_TAPE_RUN_WORKDIR=/tmp/demo ./docker/run.sh play.sh samples/fibonacci 2
----

In that case, the wrapper mounts `/tmp/demo` as `/workspace`, and generated files are created there.

You can also run through the wrapper:

[,bash]
----
./docker/run.sh play.sh samples/fibonacci
./docker/run.sh play.sh samples/fibonacci 2
./docker/run.sh rewind.sh samples/fibonacci
./docker/run.sh rewind.sh samples/fibonacci 2
----

== Via Development Containers

The repository includes a standard Dev Container definition based on an
official Dev Containers base image plus the official Docker-in-Docker feature.
Its purpose is operational, not architectural: it lets you use the existing
project commands inside a Development Container or a GitHub Codespace without
changing how {BashTapeDeck} works.

The configured environment keeps the current conventions:

* the repository is mounted at `/workspaces/<repository-name>`
* the default tape workdir is `/workspaces/<repository-name>/build`
* `BASH_TAPE_RUN_WORKDIR` still overrides the default workdir
* Docker access is provided by the Dev Container Docker feature

To use it locally with Development Containers through an editor integration:

. Install a compatible Development Containers environment.
   For example, Visual Studio Code with the Dev Containers extension.
. Open this repository in that environment.
. Choose the command that reopens the project inside the Dev Container.
. Wait for the container to build and for the terminal to become available.
. Run the same project commands you would run on the host.

To use it locally through the `devcontainer` CLI:

. Install the `devcontainer` CLI.
. From the repository root, start the environment:
+
[,bash]
----
devcontainer up --workspace-folder .
----
. Open a shell inside the running Dev Container:
+
[,bash]
----
devcontainer exec --workspace-folder . bash
----
. Run the same project commands you would run on the host.

To use it remotely with GitHub Codespaces:

. Create a Codespace for this repository.
. Wait for the configured Dev Container to finish starting.
. Open the integrated terminal.
. Run the same project commands you would run locally.

In both cases, you should be able to use commands such as:

[,bash]
----
./play.sh samples/fibonacci
./rewind.sh samples/fibonacci 2
./docker/build.sh samples/fibonacci
./docker/run.sh play.sh samples/fibonacci
----

The intent is that Development Containers and Codespaces change only the
execution environment, not the {BashTapeDeck} command line or its workdir
conventions.
